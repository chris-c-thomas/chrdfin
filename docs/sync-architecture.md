# Sync Architecture

How chrdfin pulls market and macro data, persists it locally, and stays current. Read this before touching anything under `apps/desktop/src-tauri/src/sync/`, `storage/`, or the read commands in `commands/data.rs`.

---

## Why a single doc

The sync stack is the seam between three independently moving parts:

- **Providers** (Massive today; Schwab/Tradier in the post-1.0 trading roadmap, others as needed).
- **Storage** (DuckDB today; a TimescaleDB / ClickHouse swap is plausible if real-time tick storage ever lands — see § Storage backend swap path).
- **Consumers** (Tauri command handlers, the UI, the background scheduler).

The traits and tables below exist so each part can move without breaking the others.

---

## Layers

```
                    ┌─────────────────────────────────┐
   UI / commands ──▶│        Sync (orchestrator)      │◀── scheduler
                    └────────────────┬────────────────┘
                                     │
                ┌────────────────────┼────────────────────┐
                ▼                    ▼                    ▼
        DataProvider trait   MacroProvider trait    storage::*
        (Massive impl)       (Massive impl)         (DuckDB)
```

- `sync::provider` defines `DataProvider` (prices, dividends, splits, metadata, search) and `MacroProvider` (closed-enum `MacroSeriesId` series).
- `sync::massive::client::MassiveProvider` is the only concrete adapter today. Both traits land on the one struct because Massive serves both equity and macro endpoints.
- `sync::orchestrator::Sync` is the single thing the rest of the app calls. UI never reaches into providers or storage directly — it goes through `Sync::run`, `Sync::ensure_ticker`, `Sync::seed_if_needed`, or one of the `commands::data::*` read commands (which themselves go through `Sync` for any on-demand fetch).
- `sync::scheduler::spawn_background_sync` owns the long-lived background task: seed on first launch, catch-up if stale, then daily 6 PM ET incremental.

## Source-priority model

Every row written by an adapter carries a `source` column. The orchestrator's writer (`storage::*::upsert_*`) refuses to overwrite a higher-priority row with a lower-priority one. Priorities live in `storage::source::SOURCE_PRIORITY`:

| Source             | Priority |
|--------------------|----------|
| `manual_backfill`  | 100      |
| `massive`          | 50       |

A future Tiingo or Schwab adapter just adds a new row to that table at the appropriate priority. The conflict resolution is encoded as a SQL `CASE` expression generated by `storage::source::priority_case_sql` and inlined into each upsert's `ON CONFLICT WHERE` clause — no DuckDB UDF, no per-connection registration.

## Orchestrator lifecycle

`Sync::run(mode, on_progress)`:

1. Acquire `run_mutex` (Tokio `try_lock`). Returns "sync already running" if a manual or scheduled run is in flight — manual + scheduled share this mutex so they never overlap.
2. Open a `sync_log` row in `started`.
3. For each ticker in `app_settings.tracked_universe`: ensure metadata exists, then fetch prices / dividends / splits for the appropriate window. Per-ticker errors land in `summary.errors` rather than aborting the run.
4. For each `MacroSeriesId` in `DEFAULT_MACRO_SERIES`: fetch + upsert.
5. Close the `sync_log` row to `completed` with row counts.

`Sync::ensure_ticker(ticker, start, end)`:

- Per-ticker async `Mutex` from a `DashMap` keeps concurrent callers for the same symbol queued behind a single backend fetch.
- Looks up local coverage; if `latest_price_date >= end`, returns immediately. Otherwise fetches the gap.
- Used by `commands::data::get_prices` / `get_asset_metadata` for on-demand population.

`Sync::seed_if_needed`:

- Reads `app_settings.seeded`; returns immediately if `true`.
- Sets `app_settings.tracked_universe = STARTER_UNIVERSE` (only if the user hasn't already populated it).
- Runs `Sync::run(Full)`.
- Flips `seeded = true` once at least one ticker has landed — partial completion still moves the install forward; subsequent runs incrementally top up.

## Scheduler

`sync::scheduler::spawn_background_sync` returns a `JoinHandle<()>` for a detached Tokio task that runs for the app's lifetime.

**Cadence:** daily at **6 PM America/New_York** on weekdays (Sat/Sun skipped to next Monday). DST transitions handled by `chrono-tz`.

**Catch-up:** on launch, if `last_successful_sync` is missing or older than 24 hours, immediately runs `Sync::run(Incremental)` instead of waiting for the next 6 PM tick.

**Failure backoff:** failed run → 1-hour wait → retry, capped at 4 attempts before falling back to the next-day cadence.

**Pure helpers:** `next_run_at(now: DateTime<Utc>, tz: &Tz) -> DateTime<Utc>` is pure and unit-tested for weekday/weekend/DST cases.

## Free-tier vs paid-tier configuration

`MASSIVE_TIER` env var selects the runtime profile (defaults to `Free`):

| Profile  | Rate limit  | History clamp  |
|----------|-------------|----------------|
| `Free`   | 5 requests/min | last ~2 years (Massive aggregates ceiling) |
| `Paid`   | 100 requests/sec | from 1990-01-01 |

Paid-tier synonyms (`starter`, `developer`, `advanced`, `business`) all map to `MassiveTier::Paid`. Both profiles are read once at startup by `Secrets::from_env()` and threaded through `Sync::new`.

A missing `MASSIVE_API_KEY` is **not** a startup error — the app launches so the user can still browse already-cached data. The missing key surfaces as `ProviderError::MissingApiKey` only when a sync is actually attempted.

## Adding a second adapter

The post-v1.0 trading roadmap (paper / live / bot) requires broker adapters. The pattern they follow:

1. Add a `pub struct SchwabProvider { ... }` (or whatever) in `sync::<vendor>::client`. Hold whatever auth / HTTP shape that vendor needs.
2. `impl DataProvider for SchwabProvider` (or `impl MacroProvider`, or both).
3. Add the source string + priority to `storage::source::SOURCE_PRIORITY`. Choose the priority based on the vendor's data quality / freshness vs. existing sources.
4. Where the orchestrator currently holds `Arc<MassiveProvider>`, generalize to a small enum or a trait object. The choice depends on whether the adapter introduces new methods beyond what `DataProvider` covers (broker order routing, account snapshots) — those go on a separate trait.
5. Wire the new tier / API key into `Secrets::from_env()`.
6. Add wiremock-backed unit tests under `tests/<vendor>_adapter.rs` mirroring `tests/massive_adapter.rs`.

The `Sync` struct itself does not need to change for a pure-data adapter — the source-priority model handles the multi-provider write path automatically.

## On-demand fetch dedup

A subtle but important property: when the UI opens a ticker page for a symbol the user has never seen, two things happen in parallel:

1. The chart's `usePrices` hook calls `get_prices` → `Sync::ensure_ticker`.
2. The detail panel's `useAssetMetadata` hook calls `get_asset_metadata` → `Sync::ensure_ticker`.

Both calls hit the same per-ticker `Mutex` from `Sync::in_flight: DashMap<String, Arc<TokioMutex<()>>>`, so the second one waits for the first to finish, then re-reads from DuckDB instead of re-issuing an HTTP request. This is what keeps the free-tier rate budget from being burned twice on the same ticker.

## Storage backend swap path

DuckDB is the right answer for chrdfin's current shape (single-user, EOD analytics, columnar aggregations). A future need for real-time tick storage, deep options-chain history, or multi-instance shared state could reasonably push to TimescaleDB or ClickHouse. The seams that make that swap practical:

- **All SQL lives in `storage::*`.** No `conn.execute`/`conn.query_row` calls outside that module. Replacing the backend means rewriting that one module against the new driver, not chasing string SQL through the orchestrator.
- **`Database` is a single-field newtype** (`Mutex<Connection>`). Nothing in the codebase reaches past it into DuckDB-specific APIs.
- **Schema lives in one file** (`apps/desktop/src-tauri/src/schema.sql`). Translate that to the target dialect once, run it on first launch.
- **Wire format is camelCase JSON** (via `serde(rename_all = "camelCase")` on the public DTOs). The TypeScript layer doesn't care which engine's behind the IPC boundary.

If the swap ever happens, the realistic order: (1) build a `storage` trait abstraction with both backends behind it, (2) ship a migration command that copies DuckDB rows to the new backend, (3) flip the `Database` newtype's inner type. Phase 1 deliberately does **not** introduce that trait — premature abstraction without a real second backend in mind.
